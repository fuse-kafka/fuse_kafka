00010 
00011 Intercepts all writes to specified directories and send them 
00012 to apache kafka brokers.  Quite suited for log centralization.

00035 Quickstart (from sources)
00036 =========================
00037 
00038 [Here is a capture of a quickstart](http://playterm.org/r/fusekafka-quickstart-1416935569).
00039 ([Download it](http://abdessel.iiens.net/fuse_kafka/ttyrecord) - use ttyplay to view)
00040 
00041 
00042 
00043 If you want to test fuse\_kafka, using a clone of this repository.
00044 
00045 On Debian and Ubuntu, you should install the following:
00046 
00047  *  librdkafka-dev
00048  *  librdkafka1
00049  *  libzookeeper-mt-dev
00050  *  libzookeeper-mt2
00051  *  libjansson-dev
00052  *  libjansson4
00053  *  python
00054 
00055 First, build it:
00056 
00057     $ ./build.py
00058 
00059 On another terminal session, start zookeeper (this will download kafka):
00060 
00061     $ ./build.py zookeeper_start
00062 
00063 On another one, start kafka:
00064 
00065     $ ./build.py kafka_start
00066 
00067 The default configuration is conf/fuse\_kafka.properties.
00068 An important piece of configuration is fuse\_kafka\_directories:
00069 
00070     $ grep fuse_kafka_directories conf/fuse_kafka.properties -B2
00071     # directories fuse_kafka will listen to (launch script will try to
00072     # create them if they don't exist)
00073     fuse_kafka_directories=["/tmp/fuse-kafka-test"]
00074 
00075 Start fuse\_kafka using the init script:
00076 
00077     $ src/fuse_kafka.py start
00078 
00079 If you're not running as root, you might have to make 
00080 /etc/fuse.conf readable by your user (here to all users):
00081 
00082     $ chmod a+r /etc/fuse.conf
00083 
00084 And allow non-root user to specify the allow\_other option, by adding
00085 a line with user\_allow\_other in /etc/fuse.conf.
00086 
00087 If fuse\_kafka is running, you should get the following output when
00088 running:
00089 
00090     $ src/fuse_kafka.py status
00091     listening on /tmp/fuse-kafka-test
00092     fuse kafka is running
00093 
00094 In yet another terminal, start a test consumer:
00095 
00096     $ ./build.py kafka_consumer_start
00097 
00098 Then start writing to a file under the overlay directory:
00099 
00100     $ bash -c 'echo "foo"' > /tmp/fuse-kafka-test/bar
00101 
00102 You should have an output from the consumer similar to this:
00103 
00104     event:
00105         group: users
00106         uid: 1497
00107         @tags:
00108             -  test
00109         @fields:
00110              hostname: test
00111         @timestamp: 2014-10-03T09:07:04.000+0000
00112         pid: 6485
00113         gid: 604
00114         command: bash -c echo "foo"
00115         @message: foo
00116         path: /tmp/fuse-kafka-test/bar
00117         @version: 0.1.3
00118         user: yazgoo
00119 
00120 When you're done, you can stop fuse\_kafka:
00121 
00122     $ src/fuse_kafka.py stop

00125 Quota option test
00126 =================
00127 
00128 First, commment fuse\_kafka\_quota in conf/fuse\_kafka.properties.
00129 Then, start fuse kafka.
00130 
00131     $ src/fuse_kafka.py start
00132 
00133 Let's create a segfaulting program:
00134 
00135     $ cat first.c
00136     int main(void)
00137     {
00138         *((int*)0) = 1;
00139     }
00140 
00141     $ gcc first.c
00142 
00143 Then start a test consumer, displaying only the path and message\_size-added fields
00144 
00145 Launch the segfaulting program in fuse-kafka-test directory:
00146 
00147     $ /path/to/a.out
00148 
00149 A new core file should appear in fused directory.
00150 
00151 Here is the consumer output:
00152 
00153     $ SELECT="message_size-added path" ./build.py kafka_consumer_start
00154     event:
00155         message_size-added: 4096
00156         path: /tmp/fuse-kafka-test/core
00157     ...
00158     event:
00159         message_size-added: 4096
00160         path: /tmp/fuse-kafka-test/core
00161 
00162 Here we see many messages.
00163 
00164 Then, uncomment fuse\_kafka\_quota in conf/fuse\_kafka.properties and 
00165 launch the segfaulting program,
00166         
00167     $ SELECT="message_size-added path" ./build.py kafka_consumer_start
00168     event:
00169         message_size-added: 64
00170         path: /tmp/fuse-kafka-test/core
00171 
00172 This time, we only receive the first write.

00174 Event format
00175 ============
00176 
00177 We use a logstash event, except the message and command are base64 encoded:
00178 
00179     {"path": "/var/log/redis_6380.log", "pid": 1262, "uid": 0, "gid": 0,
00180     "@message": "aGVsbG8gd29ybGQ=",
00181     "@timestamp": "2014-09-11T14:19:09.000+0000","user": "root", "group":
00182     "root",
00183     "command": "L3Vzci9sb2NhbC9iaW4vcmVkaXMtc2VydmVyIC",
00184     "@version": "0.1.2",
00185     "@fields": {
00186         "first_field": "first_value",
00187         "second_field": "second_value" },
00188     "@tags": ["mytag"]}

00216 Anti hanging
00217 ============
00218 
00219 Fuse-kafka must never make your filesystem accesses hang.
00220 Although this should be considered as a major bug, this might happen
00221 since the soft is still young.
00222 You can install a script call in your crontab so that any FS hanging
00223 is umounted (the check will occur every minute).
00224 To do so on an installed instance:
00225 
00226     # service fuse_kafka cleanup
00227 
00228 To do so on a source based instance:
00229 
00230     $ ./src/fuse_kafka.py cleanup

00233 setup.sh options
00234 ================
00235 
00236 Here are available options:
00237 
00238  - `-r`: to do a remote install via ssh: `-r private_ssh_key user@host`
00239  - `-d`: download and do not install the packages, generating an archive
00240  - `-f`: install an archive already built via -d: `-f fuse_kafka.tar.bz2`
00241 
00242 For example, this will download packages on a remote server:
00243 
00244     ./setup.sh -r mykey.pem root@myserver -d
00245 
00246 This will generate an archive that will be copied locally.
00247 You can then install that archive via:
00248 
00249     ./setup.sh -f fuse_kafka.tar.bz2

00252 Networking tests
00253 ================
00254 
00255 A more realistic network setup test can be launched (as root) via:
00256 
00257     sudo ./build.py mininet
00258 
00259 This requires [mininet](http://mininet.org).
00260 
00261 This will launch kafka, zookeeper, fuse_kafka, and a consumer 
00262 on their own mininet virtual hosts with their own network stacks.
00263 fuse_kafka is running on h3 (host number three).
00264 This will launch a mininet shell.
00265 For example, if you want to try and write on fuse_kafka host, issue a:
00266 
00267     mininet> h3 echo lol > /tmp/fuse-kafka-test/xd
00268 
00269 The consumer log is available via (/tmp/kafka_consumer.log). 
00270 
00271 `quit` or `^D` will stop mininet and cleanup the virtual network.
